## 1. What “single client/secret” means in this pattern

In Option B2, the **single client/secret belongs to your auth service**, not to Cognito.

* External M2M services see **one logical OAuth client**:

  * `client_id = "service-A"`
  * `client_secret = "xyz..."`

* Internally, the auth broker has **multiple Cognito app clients**, one per user pool/region:

  * `cognito_client_id_us_east_1`
  * `cognito_client_secret_us_east_1`
  * `cognito_client_id_eu_west_1`
  * `cognito_client_secret_eu_west_1`
  * etc.

External clients never know or care about these internal Cognito credentials.
You can add/remove/rotate Cognito app clients or even add new regions without changing that external `client_id/secret`.

That is how you get the “single client/secret” story.

## 2. High-level architecture of the auth broker

Think of the broker as a **custom OAuth2 token service** that sits in front of Cognito:

* Deployed **in multiple AWS regions** (e.g., us-east-1, eu-west-1).
* Fronted by:

  * Route 53 latency-based or failover routing, or
  * AWS Global Accelerator.

Core components per region:

* **Token API** (e.g., API Gateway or ALB + ECS/Lambda):

  * Implements `/oauth2/token` for `client_credentials`.
* **Client registry** (global):

  * DynamoDB Global Table or equivalent.
  * Stores the external clients (`client_id`, hashed `client_secret`, scopes, metadata).
* **Configuration/secrets for Cognito**:

  * Mapping from region → user pool ID → app client ID/secret.
  * Stored in AWS Secrets Manager or SSM Parameter Store, replicated per region.
* **Optional cache**:

  * For caching Cognito tokens (to reduce calls) or your own issued tokens.

Because the client registry is global and the configuration is replicated, **any regional instance of the broker** can authenticate the same external client with the same `client_id/secret`.

## 3. Flow: how a single client/secret works across regions

### 3.1 Normal operation (no failure)

Assume:

* External service: `payment-worker`
* Logical external client: `client_id = payment-worker`, `secret = S1`
* Broker regions: `us-east-1`, `eu-west-1`
* Cognito user pools: `pool-USE1`, `pool-EUW1`, each with its own app client

Flow:

1. `payment-worker` calls:

   ```http
   POST https://auth.mycompany.com/oauth2/token
   Authorization: Basic base64("payment-worker:S1")
   Content-Type: application/x-www-form-urlencoded

   grant_type=client_credentials&scope=payment.write
   ```

2. DNS / Global Accelerator routes this to the **nearest healthy auth-broker** instance, say `us-east-1`.

3. Auth broker in `us-east-1`:

   * Looks up `client_id = payment-worker` in the **global client registry**.
   * Verifies the supplied `client_secret` (e.g., bcrypt hash compare).
   * Checks allowed scopes.

4. Broker decides which Cognito region to use:

   * Simple case: choose the **local region’s** pool (`pool-USE1`).
   * More advanced: health-check Cognito endpoints and choose a healthy one.

5. Broker calls Cognito `/oauth2/token` using its **own internal client credentials**:

   ```http
   POST https://cognito-idp.us-east-1.amazonaws.com/<pool-USE1>/oauth2/token
   Authorization: Basic base64("<cognito_client_id_us_east_1>:<cognito_client_secret_us_east_1>")
   grant_type=client_credentials&scope=payment.write
   ```

6. Cognito returns an **access token (JWT)**.

7. Broker verifies / inspects token (optional), then returns it to `payment-worker` as the response to `/oauth2/token`.

From `payment-worker`’s perspective:

* It only ever talks to `auth.mycompany.com`.
* It only ever uses **one** `client_id/secret`.
* It does not know or care which Cognito pool actually issued the token.

### 3.2 Region failure / failover

Suppose `us-east-1` is down.

1. DNS/Global Accelerator now routes `https://auth.mycompany.com` to `eu-west-1` broker.

2. `payment-worker` still sends:

   ```http
   client_id = payment-worker
   secret = S1
   ```

3. The broker in `eu-west-1`:

   * Uses the **same global client registry** to validate `payment-worker / S1`.
   * Chooses a healthy Cognito pool (maybe now `pool-EUW1`).
   * Calls Cognito in `eu-west-1` with `cognito_client_id_eu_west_1` / `secret`.

4. Returns a token (possibly with a different `iss`, if you pass through Cognito tokens).

No configuration change on the client. Same `client_id/secret` before and after failover.

## 4. Two variants for tokens

There are two main ways for the broker to handle tokens.

### 4.1 Variant A – Pass-through Cognito tokens (simple, but multiple issuers)

Here, your broker is just a **smart proxy**:

* It authenticates the external client.
* It calls Cognito to get a token.
* It returns Cognito’s token directly.

**Implications:**

* Your resource servers (APIs) must trust **multiple Cognito issuers**, e.g.:

  * `iss = https://cognito-idp.us-east-1.amazonaws.com/<pool-USE1>`
  * `iss = https://cognito-idp.eu-west-1.amazonaws.com/<pool-EUW1>`
* They must be able to fetch JWKS from all relevant user pools.
* Authorization logic can use standard Cognito claims (`scope`, `cognito:groups`, custom claims, etc.).

**Why this still solves “single client/secret”:**

* External clients only ever hold a single `(client_id, secret)` pair to talk to the broker.
* All Cognito complexities (multiple app clients/regions) are **abstracted away** inside the broker.
* Adding a new Cognito region or rotating Cognito secrets does not affect external clients.

### 4.2 Variant B – Broker issues its own tokens (single issuer, more work)

In this variant, the broker uses Cognito as an upstream identity provider (if needed), but it issues its **own JWTs** with a **single stable issuer**, such as:

* `iss = https://auth.mycompany.com`

Flow:

1. Broker authenticates client (`client_id`, `secret`) against its own registry.
2. Optionally, it calls Cognito to:

   * Validate something,
   * Pull group memberships or extra claims,
   * Or not use Cognito at all for “pure” M2M.
3. Broker creates its own JWT:

   * Signed by a private key stored in KMS.
   * `iss = https://auth.mycompany.com`
   * `sub = payment-worker`
   * `aud` = your APIs
   * `scope`, `tenant`, etc. as claims.
4. Broker returns this signed JWT.

Your APIs:

* Only trust one issuer (`https://auth.mycompany.com`).
* Fetch the broker’s **JWKS** from a single endpoint, regardless of region.

**Pros:**

* Clean validation story: single issuer, single JWKs URL.
* You can change or even drop Cognito later without touching clients or APIs.
* Broker can encode arbitrary authZ data.

**Cons:**

* You are now running a **custom authorization server**, with all that entails:

  * Key rotation
  * Proper JWT lifetimes and revocation strategy
  * OAuth2/OIDC semantics

In many M2M cases, this is acceptable and common, but it is extra responsibility.

## 5. How this concretely solves the “single client/secret” issue

To tie it directly back to your question:

* Without the broker, each M2M app needs **N different Cognito app clients** (or at least be reconfigured) if you want multiple Cognito regions/user pools. There is no native “global app client” in Cognito.
* With the broker:

  * You define **one canonical OAuth client per M2M service** in your own registry.
  * You issue one `client_id/secret` per service.
  * Any broker instance in any region can authenticate that service against this global registry.
  * The broker then fan-outs to **whichever Cognito app client / region** is appropriate at that moment.
  * You can change the mapping of:

    * `service → scopes`
    * `region → Cognito pool + app client`
    * token TTLs, etc.
      without pushing changes to the M2M services.

So the requirement “I want my M2M app to have a single client/secret and still be multi-region capable” is satisfied by:

* Making the **single client/secret belong to your broker**, not to Cognito.
* Letting the broker manage the messy part: multiple Cognito user pools and app clients across regions.

Understood — here is a **clear, detailed sketch specifically for Option 4.1: *Pass-Through Cognito Tokens***.

This version does **not** have the broker issue its own JWTs.
Instead, it **returns Cognito’s tokens directly**, but still allows you to present a **single client_id/secret** to external M2M callers while internally managing multiple Cognito app clients across multiple regions.

---

# 1. What this pattern achieves

You get:

* **One logical OAuth2 client_id/secret** *per M2M application*.
* **Multiple Cognito user pools and app clients** behind the scenes (one per region).
* Automatic failover: the broker chooses whichever Cognito region is healthy.
* Unmodified external M2M clients during failover or regional expansion.

You do **not** get:

* A single issuer.
  Each region’s Cognito pool still has its own `iss` and JWKS.

Your APIs must trust **multiple Cognito issuers**.

# 2. High-level architecture (Pass-Through Mode)

```text
        External M2M Client
            client_id=payment-worker
            client_secret=S1
                    |
                    v
        https://auth.mycompany.com
       (Route53 or Global Accelerator)
                    |
    +---------------+--------------+
    |                              |
us-east-1                      eu-west-1
API Gateway                   API Gateway
   |                                |
Broker Lambda/ECS            Broker Lambda/ECS
   |                                |
Cognito Pool A                Cognito Pool B
(AWS Region A)                (AWS Region B)
(App Client A)               (App Client B)
```

Key point:
**The broker picks the Cognito pool at runtime** and calls its `/oauth2/token` with the *Cognito* app client's credentials (not the external client’s).

The external client always uses only:

```
client_id = payment-worker
client_secret = S1
```

# 4. Broker logic (Pass-Through)

### Step 1: Authenticate the caller against **your** client registry

* DynamoDB Global Table:

  * Key: `client_id`
  * Attributes: `secret_hash`, `allowed_scopes`, etc.
* Validate Basic Auth.
* Enforce allowed scopes.

External clients do **not** have Cognito credentials.
They authenticate only to *your broker*.

---

### Step 2: Select which Cognito region to use

Your broker implements a regional selection strategy:

**Simple strategy:**

* Use the same region the broker is running in.

**Resilient strategy:**

* Health check all Cognito token endpoints:

  ```
  https://cognito-idp.<region>.amazonaws.com/<userPoolId>/oauth2/token
  ```
* Pick the first healthy region.
* Cache health state for e.g. 30 seconds.

So if `us-east-1` Cognito is degraded, the broker in any region automatically switches to `eu-west-1` or another available region.

---

### Step 3: Broker calls Cognito `/oauth2/token`

Example:

```http
POST https://cognito-idp.eu-west-1.amazonaws.com/<PoolB>/oauth2/token
Authorization: Basic base64("<BrokerClientID_B>:<BrokerClientSecret_B>")
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&scope=payment.write
```

This uses **Cognito app-client credentials specific to that region**.

You store these in AWS Secrets Manager:

```
COGNITO[region].CLIENT_ID
COGNITO[region].CLIENT_SECRET
COGNITO[region].POOL_ID
```

---

### Step 4: Cognito returns a standard access token (JWT)

Cognito responds:

```json
{
  "access_token": "<JWT from Cognito>",
  "expires_in": 3600,
  "token_type": "Bearer"
}
```

The JWT’s issuer looks like:

```
"iss": "https://cognito-idp.eu-west-1.amazonaws.com/<PoolB>"
```

The broker does **no signing** in this mode — it simply passes the token through.

---

### Step 5: Broker returns Cognito’s response directly

Your broker returns the Cognito payload intact (or wrapped, if you want complete transparency).

External caller receives:

```json
{
  "access_token": "<same JWT Cognito issued>",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

This is indistinguishable from receiving it directly from Cognito.

# 5. How APIs validate tokens

Because tokens may come from any Cognito region, APIs must accept **multiple issuers**.

Your authorization layer must trust:

* `iss = https://cognito-idp.us-east-1.amazonaws.com/<PoolA>`
* `iss = https://cognito-idp.eu-west-1.amazonaws.com/<PoolB>`
* etc.

Your API validation logic should:

1. Extract the issuer from the JWT.
2. Match against an allowlist of trusted Cognito pools.
3. Fetch JWKS dynamically from:

   ```
   https://cognito-idp.<region>.amazonaws.com/<PoolId>/.well-known/jwks.json
   ```
4. Cache JWKs.

This is the **core trade-off** of this option:
You get multi-region resilience, but your APIs must understand multiple Cognito issuers.

# 6. Detailed AWS sketch (components)

### 6.1 Route 53 or AWS Global Accelerator

* Domain: `auth.mycompany.com`
* Targets: APIGW in each region.

### 6.2 API Gateway (regional)

* Exposes:

  * `POST /oauth2/token`
  * `GET /health`

### 6.3 Lambda / ECS Broker

* Reads:

  * DynamoDB global table: client registry
  * Secrets Manager: Cognito credentials for each region
* Performs:

  * Basic Auth verification
  * Scope enforcement
  * Region selection
  * Calls Cognito’s `/oauth2/token`
  * Returns Cognito’s response

### 6.4 Cognito User Pools (multiple regions)

Each region has:

* A user pool
* A broker-only app client (used only by the broker)

  * Different client_id and secret per region

### 6.5 APIs / resource servers

* Allowlist multiple issuers.
* Validate JWTs issued by Cognito in any region.

# 7. Pseudo-code for the broker (Pass-Through)

```python
def handler(event):
    client_id, client_secret = parse_basic_auth(event)

    # 1. Validate against your client registry
    record = ddb_get("AuthBrokerClients", client_id)
    if not record or not verify_secret(client_secret, record["secret_hash"]):
        return oauth_error("invalid_client")

    requested_scopes = parse_scopes(event)
    if not allowed(requested_scopes, record["allowed_scopes"]):
        return oauth_error("invalid_scope")

    # 2. Select Cognito region
    cognito_region = choose_healthy_cognito_region()

    cognito_client_id = secrets[cognito_region]["client_id"]
    cognito_client_secret = secrets[cognito_region]["client_secret"]
    cognito_token_url = f"https://cognito-idp.{cognito_region}.amazonaws.com/{pool_id}/oauth2/token"

    # 3. Call Cognito
    resp = http_post(
        cognito_token_url,
        auth=(cognito_client_id, cognito_client_secret),
        data={
            "grant_type": "client_credentials",
            "scope": " ".join(requested_scopes),
        }
    )

    # 4. Return Cognito’s response (pass-through)
    return {
        "statusCode": resp.status,
        "body": resp.body,
        "headers": {"Content-Type": "application/json"}
    }
```

# 8. Benefits and drawbacks of Option 4.1

### Benefits

* External clients use **one client_id/secret** for your whole multi-region platform.
* No token-signing logic; Cognito remains the source of truth.
* Easy to drop in additional regions mechanically.
* Broker is thin (almost a proxy + registry).

### Drawbacks

* APIs must trust **multiple Cognito issuers**, making validation more complicated.
* Tokens contain issuer-specific metadata (region affects `iss` claim).
* Latency for token issuance depends on Cognito region picked.

---

# 9. When to use Pass-Through vs Broker-Issued Tokens

Use **Pass-Through** (your requested option) when:

* You want Cognito to remain the identity authority.
* You accept multiple issuers.
* You want minimum custom logic.

Use **Broker-Issued Tokens** when:

* You need one issuer and simpler API validation.
* You want to fully abstract Cognito or possibly replace it later.
* You want custom claims or cross-region uniformity.
